# Challenge: Password Check
**Category:** Reversing  
**Difficulty:** Unknown  
**Flag:** `PP{r3v3rs3_3ng1n33r1ng_1s_fun}`  

## 📝 Description
A small compiled binary named `password_check` prompts the user for a password. If correct, it prints the flag. The challenge requires extracting the embedded password and flag from data tables inside the binary.

## 🔍 Steps Taken

### 1. Initial Binary Interaction

```bash
chmod +x password_check
./password_check
```

Behavior:

```
I forgot my password, can you help me remember?
> test
Wrong Password!
```

So the password is stored inside the binary rather than hashed or derived from user input.

### 2. Static Analysis

I examined the decompilation of the main function (provided in challenge). Key operations:

- A 32-byte buffer is filled via:

  ```c
  local_58[lVar6] = (&DAT_00116580)[lVar6] ^ (&DAT_001165a0)[lVar6];
  ```

  → This initializes the **password** as XOR of two 32-byte arrays.

- After correct input, another XOR loop:

  ```c
  *psVar8 = (byte)*psVar8 ^ *pbVar9;  // XOR with DAT_00116560
  ```

  → This transforms the password into the **flag**.

Therefore:

```
password = A XOR B
flag     = password XOR K
```

Where:
- A = bytes at DAT_00116580  
- B = bytes at DAT_001165a0  
- K = bytes at DAT_00116560  

All arrays are 32 bytes.

### 3. Mapping Virtual Addresses to File Offsets

Using `readelf -S password_check`, I identified `.rodata` starts at file/VMA offset `0x16000`.

Addresses from the decompiler:

- `0x00116560` → file offset `0x16560`
- `0x00116580` → file offset `0x16580`
- `0x001165a0` → file offset `0x165a0`

These offsets are correct because the binary’s load base is `0x00100000`.

### 4. Extracting the Data & Reconstructing the Password and Flag

Python recovery script:

```bash
python3 - << 'EOF'
import os

bin_path = "password_check"
with open(bin_path, "rb") as f:
    data = f.read()

def get_bytes(addr):
    base = 0x100000
    vma = addr - base
    return data[vma:vma+0x20]

key = get_bytes(0x00116560)
a   = get_bytes(0x00116580)
b   = get_bytes(0x001165a0)

password_bytes = bytes(x ^ y for x, y in zip(a, b))
flag_bytes     = bytes(p ^ k for p, k in zip(password_bytes, key))

print("Password:", password_bytes.decode())
print("Flag:", flag_bytes.decode().strip("\x00"))
EOF
```

Output:

```
Password: th1s_1s_my_sup3r_s3cur3_p4ssw0rd
Flag: PP{r3v3rs3_3ng1n33r1ng_1s_fun}
```

### 5. Verifying

```bash
./password_check
th1s_1s_my_sup3r_s3cur3_p4ssw0rd
```

The program prints the flag.

## 🏁 Flag
```
PP{r3v3rs3_3ng1n33r1ng_1s_fun}
```

# Challenge: fileserver  
**Category:** Misc  
**Flag 1:** `PP{h4_h4___c14551c::WtG8SYDjmSkG}`  
**Flag 2:** *Not obtained*

## 📝 Description
A custom HTTP file server written in C exposes glibc documentation. The challenge states that:

1. `/home/app/flag` contains a warmup flag.
2. `/home/app/readflag` is a SUID-style binary that should produce a second flag, but must be executed on the server.

Only HTTP access is allowed.

## 🔍 Steps Taken

### 1. Checking for Path Traversal

```bash
curl -v --path-as-is "http://10.80.14.17:8080/../../../../home/app/flag"
```

Output:

```
PP{h4_h4___c14551c::WtG8SYDjmSkG}
```

### 2. Attempting Remote Execution

Downloaded `readflag`:

```bash
curl --path-as-is \
"http://10.80.14.17:8080/../../../../home/app/readflag" -o readflag
chmod +x readflag
```

Local execution fails because it expects `/root/flag`.

### 3. Analyzing the Server Binary

Disassembly shows:

- No `exec*`, `system`, `popen`, `posix_spawn`
- Only file-read code paths

Thus: **no remote execution possible**.

### 4. `/proc` Attempts

Tried:

- `/proc/self/fd/` redirections
- `%00` null-byte tricks
- Wildcards  
- Shell-like expansions

Server ignores all except direct file reads.

## 🏁 Result:

- **Flag 1 obtained**  
- **Flag 2 unobtainable with given constraints**

# Challenge: Coin Flip  
**Category:** Blockchain  
**Flag:** `PP{1_c4n_533_y0ur_futur3::f371ePwviQ5n}`  

## 📝 Description
The coin flip depends on:

```
uint256(blockhash(block.number - 1)) % 2
```

In per-user blockchain instances, each transaction mines a block deterministically → randomness is predictable.

## 🔍 Steps Taken

### 1. Goal

Drain 100 ETH → < 10 ETH.

Max bet = 5 ETH.  
Each win = +1 ETH profit.

So ~90 predictable wins required.

### 2. Strategy

Loop:

- Query previous block hash
- Compute parity
- If even → place 5 ETH flip  
- If odd → send 0 ETH tx to advance block

### 3. RPC Instability

RPC regularly dropped connections:

```
ECONNRESET
missing response
socket hang up
```

But confirmed that on-chain state persisted.

### 4. Final State

After enough runs:

```
Chal balance: 9 ETH  
isSolved(): true
```

### 5. Flag Retrieval

```bash
nc 10.80.14.94 31337
3
```

## 🏁 Flag
```
PP{1_c4n_533_y0ur_futur3::f371ePwviQ5n}
```

# Challenge: Unfair Trade  
**Category:** Blockchain  
**Flag:** `PP{d1dn7_533m_7h47_unf41r_70_m3::NKFZ7f5JmTWz}`

## 📝 Description
A “trade” function implements complex math using Solidity 0.7.x with **unchecked arithmetic**. Fee calculation can underflow, then get capped to 150 ETH.

## 🔍 Steps Taken

### 1. Pump Phase

Observation:

- Sending 3 ETH yields 2.7 ETH output.
- Contract gains +0.3 ETH each time.

Performed **214 pump trades** → contract grows to 164.2 ETH.

### 2. Drain Phase

Send a special amount (35.2 ETH):

- Fee > 0.9×amount → subtraction underflows  
- Wrapped value → > 150 ETH → capped to 150 ETH payout  
- Net contract balance becomes:  
  `164.2 + 35.2 − 150 = 49.4 ETH`  

Less than 50 ETH → solved.

### 3. Resume Logic for Unstable RPC

The solver inferred pump count from contract balance and resumed operations automatically.

### 4. Final Output

```
Final Chal balance: 49.4 ETH
isSolved(): true
```

### 5. Flag

```bash
nc 10.80.14.108 31337
3
```

## 🏁 Flag

# Challenge: A Window to the Past  
**Category:** Forensics / Misc  
**Flag:** *Recovered during CTF but not included here*

## 📝 Description
The challenge provided a series of narrow vertical image slices created by a strip-cut shredder. The job was to reconstruct the original printed page—containing the flag—by recombining the strips manually.

## 🔍 Steps Taken

### 1. Downloading the Strips
All the provided image files were saved locally. Each represented a vertical slice of the original note.

### 2. Manual Reconstruction in MS Paint
To solve this quickly during the live event:

- Opened MS Paint
- Created a large blank canvas
- Imported each strip as a separate layer
- Used pixel nudging to align edges
- Watched for letter fragments, kerning, spacing, and continuity
- Adjusted brightness and positioning until text lined up

This process is identical to reassembling shredded paper manually.

### 3. Recovering the Flag
Once the full image was reassembled, the printed message—including the flag—became readable.

I successfully retrieved the flag during the competition.

### 4. Why the Flag Is Not Included Here
I no longer have the original strips, and I did not reconstruct the document again for this retrospective write-up.  
Therefore:

> **The flag is intentionally omitted, but was fully recovered during the event.**

## 🏁 Result
*Flag recovered at event time, but not included in this write-up.*
